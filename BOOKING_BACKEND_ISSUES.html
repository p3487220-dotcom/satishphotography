Problem #1: SQL Placeholder Mismatch

PostgreSQL Placeholders Used with SQLite Database

What Went Wrong:

The booking API was using PostgreSQL-style SQL placeholders ($1, $2, $3) but the actual database was SQLite, which uses question mark placeholders (?). This caused a critical mismatch during parameter binding.

📁 src/app/api/booking/route.ts (Line 17)
const bookingId = await db.query( ` INSERT INTO bookings (name, phone, package, date, time, location, guests, requests) VALUES ($1, $2, $3, $4, $5, $6, $7, $8) RETURNING id `, [data.name, data.phone, data.package, data.date, data.time, data.location, data.guests, data.requests] );
[Booking API] Unhandled error: RangeError: Too many parameter values were provided at SqliteAdapter.query (src\lib\db.ts:63:25) at DatabaseClient.query (src\lib\db.ts:133:24) at saveBooking (src\app\api\booking\route.ts:17:27) at POST (src\app\api\booking\route.ts:138:29) 61 | const insertSql = text.replace(/\s+RETURNING\s+id\s*$/i, ""); 62 | const stmt = this.db.prepare(insertSql); > 63 | const info = stmt.run(...params); | ^ POST /api/booking 500 in 1119ms
Solution: Automatic Placeholder Conversion
📁 src/lib/db.ts

Added a conversion function to automatically transform PostgreSQL placeholders to SQLite syntax:

private convertPostgresToSqlite(text: string): string { // Convert PostgreSQL $1, $2, ... placeholders to SQLite ? placeholders let result = text; let placeholder = 1; while (result.includes(`$${placeholder}`)) { result = result.replace(`$${placeholder}`, "?"); placeholder++; } return result; } async query(text: string, params: unknown[] = []): Promise<QueryResult> { const normalized = text.trim().replace(/\s+/g, " ").toUpperCase(); const sqliteText = this.convertPostgresToSqlite(text); // ✅ Convert placeholders if (normalized.startsWith("SELECT")) { const stmt = this.db.prepare(sqliteText); return { rows: stmt.all(...params) as unknown[] }; } if (normalized.startsWith("INSERT")) { const insertSql = sqliteText.replace(/\s+RETURNING\s+id\s*$/i, ""); const stmt = this.db.prepare(insertSql); const info = stmt.run(...params); // ... rest of logic } }

Problem #2: Email Failures Blocking Bookings

Email Validation Errors Crashing API Responses

What Went Wrong:

When email credentials were invalid or unavailable, the sendEmail() function would throw an error, which was not caught. This caused the entire POST request to fail with a 500 error, even though the booking data was already saved successfully.

[Booking API] Unhandled error: Error: Invalid login: 535-5.7.8 Username and Password not accepted at sendEmail (src\app\api\booking\route.ts:45) at POST (src\app\api\booking\route.ts:138:29) code: 'EAUTH', response: '535-5.7.8 Username and Password not accepted...' responseCode: 535, command: 'AUTH PLAIN' } POST /api/booking 500 in 14.6s
Solution: Non-Blocking Email with Error Handling

Two changes were made:

❌ Before (Blocking)

async function sendEmail(data, bookingId) {
  const transporter = nodemailer.createTransport({...});
  
  await transporter.sendMail({
    from: ...,
    to: ...,
    subject: ...,
    text: plaintext,
    html,
  });
  // ❌ If error occurs here, entire request fails
}

// In POST handler:
await sendEmail(body, bookingId);
return Response.json(...);

✅ After (Non-Blocking with Error Handling)

async function sendEmail(data, bookingId) {
  try {
    const transporter = nodemailer.createTransport({...});
    
    await transporter.sendMail({...});
  } catch (emailError) {
    // ✅ Catch errors and log without crashing
    console.warn("[Booking] Failed:", 
      emailError.message);
  }
}

// In POST handler:
// ✅ Send email without blocking response
sendEmail(body, bookingId).catch((err) => {
  console.warn("[Booking] Email failed:", 
    err.message);
});

return Response.json({ 
  success: true, 
  bookingId 
});

📋 Summary of Changes

  • File: src/lib/db.ts
    • Added convertPostgresToSqlite() method
    • Converts PostgreSQL $N placeholders to SQLite ? placeholders
    • Applied to all SQL operations (SELECT, INSERT, UPDATE, DELETE)
  • File: src/app/api/booking/route.ts
    • Wrapped sendEmail() in try-catch block
    • Made email sending non-blocking (fire-and-forget pattern)
    • Email failures now logged but don't crash the booking
  • Result: Booking API now works reliably
    • 200 OK responses for successful bookings
    • Booking data saved even if email fails
    • No more "unexpected HTML response" errors
